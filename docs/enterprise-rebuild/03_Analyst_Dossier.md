# Досье для Аналитика

## Цель
- Преобразовать продуктовое видение в измеримые бизнес-требования и формальные системные спецификации для реализации.

## Роль в программе
- Аналитик объединяет бизнес-анализ и системный анализ в едином контуре.
- Отвечает за связность: `Business Goal -> BR -> FR/NFR -> контракты -> тесты`.
- Снижает разрывы между бизнес-ценностью, архитектурными ограничениями и реализацией.

## Бизнес-домен (по текущему продукту)
- AI ассистент и сценарные рекомендации.
- Счета, платежи, задолженности.
- Обращения и их жизненный цикл.
- Звонки (пропущенные, сводки, follow-up).
- Настройки, onboarding, персонализация.

## Персоны
- Owner (ИП/директор): контроль денег, рисков и операций.
- Office manager: исполнение рутинных действий по запросам.
- Finance operator: работа со счетами, неоплатами, сверками.
- Support coordinator: обращения и контроль SLA.

## Бизнес-требования и правила
- BR-01: Пользователь получает практическую подсказку в пределах 1-2 экранов для ключевых операций.
- BR-02: Критичные операции имеют детерминированный путь без зависимости от живого ИИ.
- BR-03: Для каждого ключевого сценария доступны метрики результата.
- BR-04: Ошибки AI не блокируют завершение бизнес-задачи.
- BR-05: Продукт поддерживает ролевой UX для B2B-контекста.
- Для критичных финансовых действий обязательна явная user confirmation.
- Для живого ИИ нужна политика резервного сценария при низкой надежности.
- Влияющие на платежи изменения требуют audit trail.

## Функциональные требования (FR)
- FR-01: Аутентификация и сессия с ролевым доступом.
- FR-02: Ассистент обрабатывает запросы через конвейер интент/маршрутизатор/политики.
- FR-03: Сценарии счетов: список, фильтр, детализация, статус, действия.
- FR-04: Сценарии обращений: создание, фильтрация, просмотр деталей, статусы.
- FR-05: Сценарии звонков: список пропущенных, детали, статистика.
- FR-06: Настройки и кастомизация интерфейса.
- FR-07: История и телеметрия взаимодействий без компрометации данных.

## Нефункциональные требования (NFR)
- Доступность: 99.9% для ключевого API.
- P95-задержка ассистента: детерминированный <= 1.2s, живой ИИ <= 4.0s, резервный сценарий <= 1.5s.
- Безопасность: базовый профиль OWASP ASVS; секреты не попадают в клиентский бандл.
- Масштабируемость: горизонтальный масштаб stateless API.
- Наблюдаемость: трассировка, структурированные логи, бизнес-метрики.

## Доменные контракты и API-контур
- Invoice: `id`, `amount`, `currency`, `status`, `dueDate`, `billingPeriod`, `source`.
- Appeal: `id`, `title`, `category`, `status`, `createdAt`, `owner`, `history`.
- Call: `id`, `direction`, `missed`, `time`, `counterparty`, `summary`, `recordRef`.
- AssistantMessage: `id`, `role`, `text`, `intent`, `sourceType`, `confidence`, `actions`.
- `POST /assistant/query`, `GET /invoices`, `GET /invoices/{id}`, `GET/POST /appeals`, `GET /appeals/{id}`, `GET /calls/missed`, `GET /calls/{id}`.

## Модель ошибок
- Единый формат ошибки: `code`, `message`, `details`, `traceId`, `recoverable`.
- Политика повторов, таймаутов и circuit-breaker для внешних AI-провайдеров.

## Трассировка и приемка
- Для каждого FR фиксировать связанный BR, API-контракт, сущности данных, экран/компонент и тест-кейсы.
- Бизнес-приемка: ценность доставлена, сценарий завершен, инструментирование KPI готово.
- Операционная приемка: поддержка готова, обучающие материалы готовы, откат понятен.

## Чеклист аналитика (до разработки)
- BR/FR/NFR утверждены и непротиворечивы.
- API-контракты и словарь данных версионированы.
- Правила валидации и каталог ошибок готовы.
- Требования безопасности и приватности включены в спецификации.
- Трассировка до тестов и мониторинга заполнена.

## Шаблон для Confluence (страница аналитической спецификации)
### 1. Контекст и область
- Бизнес-проблема, стейкхолдеры, в области/вне области.

### 2. Требования
- BR, FR, NFR, ограничения и исключения.

### 3. Модель данных и API-контракты
- Сущности, связи, ограничения, запрос/ответ, ошибки.

### 4. Потоки и состояния
- Основной поток, поток исключений, ключевые диаграммы последовательностей/состояний.

### 5. Трассировка и приемка
- BR -> FR -> тесты, критерии бизнес- и операционной приемки.

### 6. Риски
- Риск, влияние, меры снижения, владелец.

## Первые обязательные артефакты
- BRD/SRS v1 (единый пакет).
- Capability map и Business Process Catalog.
- Спецификация API v1 (черновик OpenAPI).
- Каталог ошибок и словарь данных.
- Матрица трассировки требований.
- KPI dictionary и impact map по релизным волнам.
