# Планирование на базе способностей (CBP): BI -> BC -> TC

## Цель документа
- Дать управляемую CBP-декомпозицию для перехода демо-продукта в промышленную эксплуатацию.
- Связать бизнес-инициативы (BI), бизнес-способности (BC) и технические способности (TC).

## Правила декомпозиции
- `BI` (Business Initiative): целевая бизнес-инициатива, привязанная к KPI/OKR.
- `BC` (Business Capability): способность бизнеса стабильно выполнять класс задач.
- `TC` (Technical Capability): конкретная техническая способность, реализующая BC.

## BI-1: Управление финансовыми операциями клиента
### KPI/результат
- Снижение времени до оплаты на 20%.
- Рост self-service завершения финансовых операций до 70%.

### BC-1.1: Прозрачность статуса счетов
#### TC
- TC-1.1.1: Каноническая доменная модель `Invoice` и статусов.
- TC-1.1.2: API `GET /invoices`, `GET /invoices/{id}` с фильтрацией/сортировкой.
- TC-1.1.3: FE-компоненты списка/детали счета с состояниями loading/error/empty.
- TC-1.1.4: Инструментация событий просмотра и действий по счетам.

### BC-1.2: Быстрое действие по неоплаченным позициям
#### TC
- TC-1.2.1: Витрина неоплаченных счетов с приоритизацией по сроку/риску.
- TC-1.2.2: Безопасный action-flow оплаты/подтверждения.
- TC-1.2.3: Audit trail для финансовых действий.
- TC-1.2.4: Нотификация о критичных сроках.

### BC-1.3: Аналитика и рекомендации по платежам
#### TC
- TC-1.3.1: Детерминированные сценарии аналитики в ассистенте.
- TC-1.3.2: Политика AI-резервирования при недоступности провайдера живого ИИ.
- TC-1.3.3: Метрики `prompt -> financial action completed`.

## BI-2: Управление обращениями и SLA
### KPI/результат
- Снижение времени resolution обращений на 30%.
- Прозрачность SLA/статусов для 100% активных обращений.

#### TC

- TC-2.1.2: API списка/детали/создания обращений.
- TC-2.1.3: Фильтры по статусам (active/in_work/done/rejected).
- TC-2.1.4: Full-text поиск по полям обращения.

### BC-2.2: Предсказуемое создание и маршрутизация обращения
#### TC
- TC-2.2.1: Форма создания обращения с валидацией.
- TC-2.2.2: Категоризация и бизнес-правила маршрутизации.
- TC-2.2.3: SLA timers + статусные переходы.
- TC-2.2.4: Уведомления и подтверждения пользователю.

### BC-2.3: Помощь ассистента в работе с обращениями

- TC-2.3.2: Action cards для создания/открытия обращений.
- TC-2.3.3: Логика безопасных резервных ответов.

## BI-3: Управление коммуникациями и звонками
### KPI/результат
- Снижение потерь лидов из-за пропущенных звонков на 25%.
- Рост доли follow-up действий после пропущенных звонков до 60%.

### BC-3.1: Оперативная обработка пропущенных звонков
#### TC
- TC-3.1.1: Модель `Call` и журнал пропущенных.
- TC-3.1.2: API `GET /calls/missed`, `GET /calls/{id}`.
- TC-3.1.3: FE-виджет приоритетных пропущенных.
- TC-3.1.4: Навигационный flow в детали звонка.

### BC-3.2: Аналитика по звонкам для принятия решения
#### TC
- TC-3.2.1: Weekly stats виджет и сценарные follow-up.
- TC-3.2.2: Метрики по пропускам/длительности/конверсии.
- TC-3.2.3: Экспорт/отчетность по звонкам.

### BC-3.3: Интеграция ассистента в процессы звонков
#### TC
- TC-3.3.1: Intent routing для call-related запросов.
- TC-3.3.2: Deterministic flow “звонки за неделю”.
- TC-3.3.3: Контролируемый живой ИИ с guard-механизмом надежности.

## BI-4: Надежный AI-ассистент для бизнес-операций
### KPI/результат
- 95% запросов ассистента завершаются полезным outcome.
- Снижение соотношения ошибок живого ИИ к резервным ответам до целевого порога.

### BC-4.1: Предсказуемая обработка запросов
#### TC
- TC-4.1.1: Конвейер `валидация -> интент -> политика -> резолвер`.
- TC-4.1.2: Каталог интентов и версионирование правил.
- TC-4.1.3: Policy engine для критичных операций.

### BC-4.2: Надежная работа живого ИИ
#### TC
- TC-4.2.1: Multi-provider routing (Gemini/Together/OpenRouter/Groq/Grok).
- TC-4.2.2: Таймауты/retry/circuit-breaker.
- TC-4.2.3: Fallback orchestration и пользовательская объяснимость.
- TC-4.2.4: Сбор метрик качества/стоимости AI.

### BC-4.3: Безопасность и соответствие
#### TC
- TC-4.3.1: Исключение секретов из client runtime.
- TC-4.3.2: Guardrails для небезопасного контента.
- TC-4.3.3: Audit logging чувствительных действий.

## Матрица приоритизации BI (рекомендация)
- Волна 1 (MVP в продуктиве): BI-1, BI-2, BI-4, BI-5.
- Волна 2: BI-3, BI-6.
- Волна 3: расширенная аналитика/автоматизация и кросс-доменные оптимизации.
