# Досье для бэкенд-команды

## Цель
- Спроектировать бэкенд-ядро продукта для надежной интеграции доменных данных и оркестрации AI.

## Ответственность бэкенда
- Авторизация/сессия и контроль доступа.
- Доменные API: счета, обращения, звонки.
- API оркестрации AI с политиками и резервным сценарием.
- Интеграционные адаптеры к корпоративным системам.
- Аудит, логирование, мониторинг и контуры управления.

## Целевая топология сервисов
- API-шлюз / BFF.
- Сервис оркестрации ассистента.
- Адаптер биллингового сервиса.
- Адаптер сервиса обращений.
- Адаптер аналитики звонков.
- Сервис уведомлений и workflows (этап 2+).

## Принципы проектирования API
- Подход contract-first (OpenAPI).
- Политика версионирования (`/v1`, матрица совместимости).
- Стандартизированный формат ошибки.
- Correlation ID в каждом запросе.

## Оркестрация ассистента (production-паттерн)
- Конвейер:
- валидация,
- определение интента,
- проверка политик,
- детерминированный резолвер,
- маршрутизация к live-провайдеру,
- проверка безопасности и надежности,
- компоновка резервного ответа.
- Сохранение телеметрии на каждом этапе.

## Данные и хранение
- Ключевые сущности:
- user/session/role,
- invoice,
- appeal,
- сводка записи звонка,
- журнал взаимодействия с ассистентом.
- Таблицы аудита для критичных действий.
- Ключи идемпотентности для изменяющих операций.

## Требования безопасности
- Ключи провайдеров не должны находиться в публичном клиентском runtime.
- RBAC + ABAC для чувствительных операций.
- Санитизация входных данных и ограничения частоты запросов.
- Классификация PII и маскирование в логах.

## Паттерн устойчивости
- Бюджеты таймаутов для каждого внешнего зависимого сервиса.
- Повторы с backoff при временных сбоях.
- Circuit breaker для нестабильных внешних систем.
- Плавная деградация при сбоях AI/live.

## Целевые показатели производительности
- P95 чтения API <= 500ms.
- P95 записи API <= 900ms.
- P95 оркестрации ассистента <= 1500ms (детерминированный путь).

## Стратегия бэкенд-тестирования
- Модульные тесты: доменные правила/политики.
- Контрактные тесты: внешние адаптеры.
- Интеграционные тесты: границы сервисов.
- E2E API-сценарии для критичных бизнес-потоков.
- Тесты хаоса и отказов для внешних зависимостей.

## Критерии готовности для бэкенда
- OpenAPI обновлен и прошел ревью.
- Хуки наблюдаемости подключены.
- Проверки безопасности пройдены (базовый профиль SAST/DAST).
- Нагрузочные тесты для новых критичных эндпоинтов выполнены.
- Обновления runbook включены.
