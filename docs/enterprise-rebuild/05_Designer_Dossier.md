# Досье для дизайнера (продуктовый/UI/UX)

## Цель
- Перевести текущий демо-интерфейс в промышленно-готовую дизайн-систему и пользовательские потоки.

## Дизайн-принципы
- Мобильный приоритет и ясность для B2B-аудитории.
- Низкая когнитивная нагрузка для операционных задач.
- UX, ориентированный на действие: путь до целевого действия <= 3 шага.
- Паритет тем: светлая/темная темы без потери контраста и доступности.
- Прозрачность AI-UX: понятные источники ответа и причины перехода на резервный сценарий.

## Инвентаризация поверхностей продукта
- Онбординг и авторизация.
- Главный экран ассистента и чат.
- Список и детализация счетов.
- Список и детализация обращений.
- Список и детализация пропущенных звонков.
- Настройки, кастомизация, подписка и FAQ.

## UX-требования
- Единый паттерн действий: `инсайт -> решение -> действие`.
- Для AI-ответа всегда видны:
- источник (детерминированный сценарий / live / резервный сценарий),
- доступные последующие действия.
- Критичные действия имеют явное подтверждение.
- Ошибки и резервные ответы подаются как восстанавливаемое состояние, а не тупик.

## Область дизайн-системы
- Основы:
- цветовые токены,
- типографическая шкала,
- отступы, радиусы, уровни возвышения,
- правила иконографии.
- Компоненты:
- карточки, чипсы, кнопки действий, панели ввода, строки списка, статусные бейджи, модальные окна, тосты.
- Состояния взаимодействия:
- по умолчанию / наведение / нажатие / отключено / загрузка / скелетон.

## Базовые требования доступности
- WCAG 2.1 AA для текста и компонентов.
- Клавиатурная навигация для веба.
- Метки для скринридера на интерактивных элементах.
- Контраст и фокус-состояния в обеих темах.

## UX-потоки (обязательные)
- F1: Запрос -> ответ ассистента -> переход в счет -> детализация.
- F2: Пропущенные звонки -> детали -> последующее действие.
- F3: Обращения -> создание -> отслеживание статуса.
- F4: Ошибка live AI -> резервный сценарий -> завершение задачи альтернативным путем.

## Критерии дизайн-QA
- Визуальная консистентность между экранами.
- Стабильность компонентов на разных размерах экрана.
- Проверка состояния переполнения/длинного текста.
- Проверка пустых состояний/ошибок/загрузки.

## Требования к передаче в разработку
- Макеты в Figma с автолейаутом и токенами.
- Спецификация компонентов с матрицей состояний.
- Редлайны и примечания по адаптивному поведению.
- Рекомендации по текстам (тон, длина, формат).

## Риски дизайна
- Перенос демо-решений без унификации компонентной библиотеки.
- Недостаточно проработанные состояния ошибок.
- Неполная проработка темной темы.
