# Структура Jira: эпик -> стори -> задачи (по ролям)

## Цель документа
- Дать готовую структуру Jira для запуска поставки.
- Формализовать стори в формате: **«Я как ..., хочу ..., чтобы ...»**.
- На каждую стори задать задачи по ролям реализации.

## Принципы
- Эпик = крупная бизнес-ценность (обычно соответствует BI/части BI).
- Стори = проверяемый пользовательский результат.
- Задача = работа конкретной роли по реализации стори.
- У каждой стори: AC, приоритет, оценка, зависимости, владелец.

## ЭПИК-1: Финансовые операции и счета

### СТОРИ-1.1
- Я как владелец бизнеса, хочу видеть список счетов с текущими статусами, чтобы быстро понимать финансовое состояние компании.
#### Задачи по ролям
- PO: уточнить AC и KPI стори.
- BA: описать бизнес-правила статусов и сортировки.
- SA: специфицировать API-контракт и ошибки.
- Solution Architect: проверить соответствие целевой архитектуре.
- System Architect: определить NFR и мониторинг API.
- Дизайнер: макет списка, фильтров, пустых состояний и ошибок.
- Фронтенд: реализовать экран списка и фильтры.
- Бэкенд: реализовать эндпоинт списка счетов.
- Ручной QA: тест-кейсы и чеклист регрессии.
- Авто-QA: API + e2e-тесты списка и фильтров.
- PM: контроль сроков, рисков и зависимостей.

### СТОРИ-1.2
- Я как финансовый менеджер, хочу открывать детализацию счета, чтобы видеть сумму, срок и статус оплаты.
#### Задачи по ролям
- BA: сценарии детального просмотра и исключения.
- SA: контракт `GET /invoices/{id}`.
- Дизайнер: макет страницы деталей и состояний.
- Фронтенд: маршрут деталей и UI-состояния.
- Бэкенд: эндпоинт детализации и валидация ID.
- Ручной QA: позитив/негатив по детализации.
- Авто-QA: e2e-переход список -> детали.
- Остальные роли: проверка и governance (как в СТОРИ-1.1).

### СТОРИ-1.3
- Я как владелец бизнеса, хочу быстро видеть неоплаченные счета, чтобы оперативно снижать риск просрочки.
#### Задачи по ролям
- BA/SA: правила определения «неоплаченный».
- Дизайнер: приоритетная витрина неоплаченных.
- Фронтенд: виджет/фильтр неоплаченных.
- Бэкенд: серверная фильтрация и сортировка.
- QA (ручной/авто): тесты приоритизации и пограничных случаев.

## ЭПИК-2: Управление обращениями

### СТОРИ-2.1
- Я как офис-менеджер, хочу видеть активные обращения, чтобы контролировать, что находится в работе.
#### Задачи по ролям
- BA: определить бизнес-статусы и фильтры.
- SA: контракт списка обращений, пагинация и фильтрация.
- Дизайнер: список обращений и бейджи статусов.
- Фронтенд: экран списка и фильтры.
- Бэкенд: эндпоинт списка и фильтрации.
- Ручной QA: кейсы статусов и поиска.
- Авто-QA: API/e2e-покрытие.

### СТОРИ-2.2
- Я как пользователь, хочу создавать новое обращение, чтобы быстро фиксировать проблему и получать поддержку.
#### Задачи по ролям
- BA: поля формы и правила обязательности.
- SA: контракт создания обращения и валидация.
- Дизайнер: форма создания и состояния успеха/ошибки.
- Фронтенд: форма и валидация.
- Бэкенд: эндпоинт создания и аудит.
- QA: smoke/regression по созданию.

### СТОРИ-2.3
- Я как владелец бизнеса, хочу видеть историю изменений обращения, чтобы понимать прогресс решения.
#### Задачи по ролям
- SA/Бэкенд: модель истории и эндпоинт.
- Фронтенд/Дизайнер: UI ленты истории.
- QA: тесты сортировки и полноты истории.

## ЭПИК-3: Пропущенные звонки и коммуникации

### СТОРИ-3.1
- Я как менеджер, хочу видеть список пропущенных звонков, чтобы не терять клиентов.
#### Задачи по ролям
- BA/SA: критерии «пропущенного» и источники.
- Дизайнер: список с приоритизацией.
- Фронтенд: UI списка и переходы.
- Бэкенд: эндпоинт пропущенных звонков.
- QA manual/auto: фильтрация, сортировка, навигация.

### СТОРИ-3.2
- Я как менеджер, хочу открывать детали звонка, чтобы понять контекст и выполнить последующее действие.
#### Задачи по ролям
- SA: контракт деталей звонка.
- Фронтенд/Дизайнер: страница деталей звонка.
- Бэкенд: API детализации.
- QA: позитив/негатив сценарии.

### СТОРИ-3.3
- Я как владелец бизнеса, хочу получать недельную сводку звонков, чтобы принимать решения по работе команды.
#### Задачи по ролям
- BA: метрики сводки.
- SA/Бэкенд: агрегирующий эндпоинт статистики.
- Фронтенд/Дизайнер: виджет недельного дашборда.
- Авто-QA: e2e-сценарий открытия сводки.

## ЭПИК-4: AI-ассистент и надежность ответов

### СТОРИ-4.1
- Я как пользователь, хочу получать релевантные ответы ассистента по счетам, обращениям и звонкам, чтобы быстро закрывать операционные задачи.
#### Задачи по ролям
- BA: каталог запросов и ожидаемые результаты.
- SA: конвейер обработки и контракты assistant API.
- Solution Architect: проверка паттерна AI-оркестрации.
- Фронтенд: интеграция чата и карточек действий.
- Бэкенд: эндпоинт оркестратора и слой политик.
- Ручной QA: сценарии релевантности.
- Авто-QA: контрактные и e2e chat-gate тесты.

### СТОРИ-4.2
- Я как пользователь, хочу видеть безопасный резервный сценарий, если live AI недоступен, чтобы процесс не останавливался.
#### Задачи по ролям
- SA: спецификация ошибок и резервных сценариев.
- Фронтенд: UX резервного сценария и метки источника.
- Бэкенд: политика резервирования и переключение провайдера при сбое.
- QA: тесты отказов провайдеров.

### СТОРИ-4.3
- Я как владелец продукта, хочу видеть метрики качества ассистента, чтобы управлять качеством и приоритизацией улучшений.
#### Задачи по ролям
- PO/BA: KPI и словарь событий.
- SA: контракты телеметрии.
- Бэкенд: сбор и агрегация метрик.
- Фронтенд: отправка событий.
- Авто-QA: валидация событий.

## ЭПИК-5: Платформа, безопасность и эксплуатация

### СТОРИ-5.1
- Я как архитектор платформы, хочу централизованно управлять секретами, чтобы исключить утечки и снизить риск компрометации.
#### Задачи по ролям
- System Architect: проектирование управления секретами.
- Бэкенд: серверная работа с ключами, удаление из клиентского runtime.
- Фронтенд: очистка публичных env для критичных ключей.
- QA/Security: проверки на утечку секретов.
- PM: координация миграции ключей.

### СТОРИ-5.2
- Я как команда эксплуатации, хочу иметь мониторинг и алерты по SLO, чтобы быстро обнаруживать и устранять инциденты.
#### Задачи по ролям
- System Architect: стандарт наблюдаемости.
- Бэкенд/Фронтенд: интеграция трейсинга, метрик и логов.
- Авто-QA: синтетические проверки.
- PM: ритм ревью runbook.

### СТОРИ-5.3
- Я как руководитель проектов, хочу видеть готовность к релизу по единым гейтам качества, чтобы выпускать только готовые версии.
#### Задачи по ролям
- PM: процесс release board.
- QA Lead: критерии гейтинга и отчеты.
- PO/Architects: обязательные sign-off.
- FE/BE: выполнение технических гейтов.

## ЭПИК-6: UX, онбординг и внедрение

### СТОРИ-6.1
- Я как новый пользователь, хочу пройти понятный онбординг, чтобы быстрее начать работать с продуктом.
#### Задачи по ролям
- BA: целевые сценарии онбординга.
- Дизайнер: UX-поток и экраны.
- Фронтенд: реализация и состояния.
- QA: регрессионные тесты онбординга.

### СТОРИ-6.2
- Я как пользователь, хочу видеть консистентный интерфейс в светлой и темной теме, чтобы комфортно работать в любых условиях.
#### Задачи по ролям
- Дизайнер: спецификации паритета тем.
- Фронтенд: реализация паритета светлой/темной темы.
- Ручной QA: визуальная матрица.
- Авто-QA: скриншотные и DOM-проверки критичных экранов.

### СТОРИ-6.3
- Я как владелец продукта, хочу видеть воронки использования ключевых функций, чтобы улучшать внедрение.
#### Задачи по ролям
- PO/BA: определить воронки и KPI.
- Фронтенд/Бэкенд: события и аналитика.
- Авто-QA: валидация конвейера телеметрии.

- `Название эпика`
