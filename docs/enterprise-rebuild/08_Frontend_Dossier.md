# Досье для фронтенд-команды

## Цель
- Определить целевую фронтенд-архитектуру и инженерные стандарты для промышленной разработки.

## Текущая базовая линия (по репозиторию)
- Next.js App Router, React 18, TypeScript, Tailwind.
- Модульные экраны через общие наборы компонентов (`ai-kit`, `classic-kit`).
- Логика ассистента частично на клиенте.
- Сильная база e2e- и модульных тестов.

## Целевая фронтенд-архитектура
- Слоистая фронтенд-архитектура:
- `app`: маршрутизация и композиция,
- `features`: бизнес-модули,
- `entities`: доменные модели,
- `shared`: общий UI-набор и утилиты.
- Явное разделение представления, состояния и доступа к данным.

## Управление состоянием
- Серверное состояние: подход уровня React Query/SWR.
- Клиентское состояние: легковесное хранилище для UI/сессии.
- Запрещена бизнес-критичная логика принятия решений во view-компонентах.
- Побочные эффекты централизованы в слое use-cases/hooks.

## Стратегия доступа к API
- Типизированный API-клиент, сгенерированный из OpenAPI.
- Единая обработка ошибок и перехватчиков.
- Идемпотентные мутации там, где это необходимо.
- Повторные попытки только для безопасных операций.

## Маршрутизация и навигация
- Политика guard-маршрутов через middleware авторизации/сессии.
- Единая стратегия deep-link для ключевых сценариев.
- Явные границы загрузки и ошибок на каждый сегмент маршрута.

## Правила UI/дизайн-системы
- Приоритет библиотечных компонентов; ad hoc стили минимизируются.
- Паритет светлой и темной темы обязателен для всех UI-изменений.
- Проверки доступности встроены в процесс PR.
- Запрещен прямой хардкод бизнес-текстов без i18n/content-политики.

## Правила интеграции AI в UX
- Всегда показывать метку источника ответа.
- Детерминированные сценарии имеют приоритет для критичных операций.
- Live AI-ответ не должен блокировать выполнение ключевого потока.
- Резервный UX должен предлагать практически применимые альтернативы.

## Целевые показатели производительности
- Бюджет Core Web Vitals:
- LCP < 2.5s,
- INP < 200ms,
- CLS < 0.1.
- Разделение кода по маршрутам и функциям.
- Оптимизация изображений и ресурсов.

## Стратегия FE-тестирования
- Модульные тесты: доменные утилиты, хуки, форматтеры, guard-проверки.
- Компонентные тесты: интерактивные состояния и пограничные случаи.
- E2E: критичные пользовательские пути от ассистента к доменным действиям.
- Визуальная регрессия для изменений дизайн-системы.

## Критерии готовности для фронтенда
- Функция покрыта модульными/компонентными/e2e-тестами по применимости.
- A11y-проверки пройдены.
- Реализованы состояния ошибок/загрузки/пустых данных.
- События аналитики отправляются и валидированы.
- Нет lint/type ошибок.

- Состояния

### 3. Контракт данных
- Эндпоинты API
- Типы данных

### 4. Состояние/логика

- Серверное состояние

- Модульные
- Компонентные
- E2E

- Фича-флаги
- Мониторинг

## Стартовый фронтенд-бэклог
- Рефакторинг архитектурного каркаса (`features/entities/shared`).
- Интеграция типизированного API-клиента.
- Единая машина состояний ассистента.
- Усиление доступности и дизайн-системы.
- Хуки наблюдаемости для пользовательских путей.
