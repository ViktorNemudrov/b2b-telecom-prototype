# Досье для QA-автоматизации

## Цель
- Построить масштабируемую стратегию автотестирования для устойчивого релизного цикла.

## Текущая базовая линия
- Уже используются Playwright e2e и Vitest для модульных тестов.
- Есть сильный набор гейт-тестов для чат-сценариев.
- Требуется промышленная стандартизация пайплайнов, управления flaky и покрытия.

## Стратегия автоматизации
- Тестовая пирамида:
- модульные тесты (основа),
- API/контрактные тесты (интеграционный слой),
- E2E-тесты (критичные пользовательские пути).
- Риск-ориентированная автоматизация:
- критичные пути автоматизируются в первую очередь.
- Правило:
- ни одной новой критичной функции без автоматизации.

## Архитектура тестов
- Структура папок по доменам (`assistant`, `invoices`, `appeals`, `calls`, `auth`).
- Общие фикстуры и стабильные сборщики тестовых данных.
- Тегирование: `@smoke`, `@regression`, `@release-gate`, `@flaky-monitor`.
- Детерминированные селекторы и page object-паттерн для критичных потоков.

## Интеграция с CI/CD
- Этапы пайплайна:
- lint/typecheck,
- unit,
- API/contract,
- e2e smoke,
- e2e nightly-regression.
- Гейт качества:
- сборка блокируется при падении release-gate набора.
- Артефакты:
- HTML-отчеты,
- junit/xml,
- trace/video/screenshots.

## Политика flaky-тестов
- Ограниченный автоматический повторный запуск (максимум 1).
- Процесс карантина flaky с владельцем и сроком исправления.
- Еженедельный обзор и очистка flaky.

## Стратегия покрытия
- Карта покрытия:
- Требования -> ID автоматических тестов.
- Минимальные пороги:
- покрытие unit-тестами критичных модулей,
- покрытие e2e-тестами критичных пользовательских путей.
- Пересмотр покрытия на каждом релизе.

## Автоматизация нефункциональных проверок
- Быстрые проверки производительности (бюджеты задержек).
- Быстрые проверки безопасности (authz, типовые уязвимости).
- Быстрая автоматизация доступности на ключевых экранах.

## Управление тестовыми данными
- Версионированные стартовые наборы данных.
- Изолированные тестовые тенанты/пользователи.
- Идемпотентные setup/teardown процедуры.
- Маскирование чувствительных наборов данных.

## Наблюдаемость в тестах
- Захват `traceId`/request ID в упавших тестах.
- Приложение логов и сетевых трасс к отчетам о сбоях.
- Автопривязка падений к известным категориям дефектов.
