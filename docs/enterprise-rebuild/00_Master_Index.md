# Пакет документации для перезапуска (RU)

## Цель пакета
- Зафиксировать полный набор требований и решений для перехода от демо-продукта к промышленной реализации.
- Дать единый источник правды для продукта, поставки, дизайна, архитектуры, разработки и QA.
- Подготовить структуру в формате Markdown, готовую к переносу в Confluence без переработки.

## Контекст текущего решения
- Монорепозиторий `apps/*` + `packages/*`, две витрины: `ai-first` и `classic`.
- Технологическая база: Next.js 14, React 18, TypeScript, Tailwind, Playwright, Vitest.
- Текущее состояние: функционально богатый демо-прототип с мок-данными и частичной live-AI интеграцией.
- Ограничение демо: бизнес-критичные процессы, безопасность, эксплуатационные контуры и реальные интеграции не доведены до production-grade.

## Каталог документов (для ролей)
- `01_Product_Owner_Dossier.md` — Владелец продукта (PO).
- `02_Project_Manager_Dossier.md` — Руководитель проектов (PM).
- `03_Business_Analyst_Dossier.md` — Бизнес-аналитик.
- `04_System_Analyst_Dossier.md` — Системный аналитик.
- `05_Designer_Dossier.md` — продуктовый/UI/UX дизайнер.
- `06_Solution_Architect_Dossier.md` — solution-архитектор.
- `07_System_Architect_Dossier.md` — системный архитектор (platform).
- `08_Frontend_Dossier.md` — фронтенд-разработка.
- `09_Backend_Dossier.md` — бэкенд-разработка.
- `10_QA_Manual_Dossier.md` — ручной QA.
- `11_QA_Automation_Dossier.md` — QA-автоматизация.

## Сквозные артефакты управления качеством
- `12_Traceability_And_Gates.md` — трассировка требований, гейты качества, готовность к релизу.
- `13_CBP_BI_BC_TC.md` — планирование на базе способностей: BI -> BC -> TC.
- `14_Jira_Epic_Story_Task_Breakdown.md` — декомпозиция Jira: эпик -> стори -> задачи по ролям.

## Рекомендуемый порядок использования
- Шаг 1: Product Owner + Business Analyst фиксируют бизнес-цели, MVP и KPI.
- Шаг 2: System Analyst формирует функциональные и интеграционные контракты.
- Шаг 3: Архитектурная связка (Solution + System) закрепляет целевую архитектуру и NFR/SLO.
- Шаг 4: Дизайн + FE/BE формируют план исполнения для MVP и релизных волн.
- Шаг 5: QA (ручной + авто) формирует тест-стратегию, карту покрытия и CI-гейты качества.
- Шаг 6: PM закрепляет ритм поставки, риск-план, управление зависимостями и релизный календарь.

## Структура для Confluence
- Каждый документ содержит:
- `Цель`
- `Область`
- `Текущее / целевое состояние`
- `Требования`
- `Артефакты`
- `Процессы`
- `Риски`
- `Шаблоны`
- `Согласование`

## Матрица ответственности (RACI верхнего уровня)
- Видение/область продукта: `A=PO`, `R=BA`, `C=SA, Architect`, `I=PM, QA, Eng`.
- Архитектурные решения: `A=Solution Architect`, `R=System Architect`, `C=FE/BE Leads, SA`, `I=PO, PM, QA`.
- Требования и контракты: `A=SA`, `R=BA`, `C=FE/BE, QA`, `I=PO, PM`.
- Поставка и релиз: `A=PM`, `R=FE/BE+QA`, `C=Architects, PO`, `I=Stakeholders`.
- Качество: `A=QA Lead`, `R=QA Manual/Auto`, `C=FE/BE`, `I=PO, PM`.

## Минимальный управленческий ритм
- Еженедельный Product Review: PO + BA + PM + Leads.
- Еженедельный Architecture Board: SA + Architects + FE/BE Leads.
- Еженедельный Quality Board: QA + FE/BE + PM.
- Раз в две недели Steering Committee: PO + PM + Sponsor + Architecture.

## Критерии успеха (уровень программы)
- Выпущен производственный MVP с контролируемыми SLA/SLO.
- Все бизнес-критичные пользовательские пути покрыты автоматизацией.
- Ведется трассировка требований до тестов и мониторинга.
- Убраны демо-зависимости (локальные флаги, сценарии только на моках для критичных функций).

## Примечание по импорту в Confluence
- Загружайте каждый файл как отдельную страницу в пространстве `Product Rebuild`.
- Иерархия: `00_Master_Index` как корневая страница, остальные как дочерние.
