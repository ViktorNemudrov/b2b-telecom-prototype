# Досье для Руководителя проектов (PM)

## Цель
- Обеспечить предсказуемую поставку промышленного продукта из текущего демо-состояния.

## Модель поставки
- Метод: гибрид Agile-подхода и этапных гейтов для архитектуры/безопасности/релиза.
- Каденс: 2-недельные спринты, 1 программный инкремент = 5 спринтов.
- Роли: PO, PM, Аналитик, Солюшн-архитектор, Системный архитектор, Лиды FE/BE, Лид QA.

## Мастер-план
- Фаза 0 (4-6 недель): инициация, discovery, архитектурная базовая линия, базовая линия нефункциональных требований.
- Фаза 1 (8-12 недель): сборка MVP + усиление надежности + готовность к промышленной эксплуатации.
- Фаза 2 (8-10 недель): интеграции ключевых систем и усиление производительности/безопасности.
- Фаза 3 (rolling): оптимизация, масштабирование, расширение функций.

## Контрольные точки
- M1: PRD утвержден + набор архитектурных решений утвержден.
- M2: API/данные контракты зафиксированы (v1).
- M3: Сквозной happy-path завершен (staging).
- M4: Проверки безопасности/производительности/UAT пройдены.
- M5: Готовность к запуску в продуктив утверждена.
- M6: Выход из hypercare.

## RAID-фреймворк
- Реестр рисков: вероятность, влияние, владелец, срок, меры снижения.
- Реестр допущений: явные допущения с валидацией.
- Реестр проблем: блокирующие/неблокирующие.
- Реестр зависимостей: внутренние и внешние.

## Примеры ключевых рисков
- Неопределенный контракт интеграций биллинга/звонков.
- Вариативность провайдеров живого ИИ (задержка/ошибки/стоимость).
- Перегрузка FE-области при слабой backend-оркестрации.
- Недостаточное покрытие E2E регрессии для критичных сценариев.

## Управление качеством и релизами
- Проверка готовности к релизу обязательна для каждого релиза.
- Нельзя выпускать релиз без:
- покрытие RTM >= целевого порога,
- закрытые blocker/critical дефекты,
- базовая линия SLO + алерты.

## План коммуникаций
- Ежедневный синк команды.
- Еженедельный обзор поставки: PM + лиды + Лид QA.
- Еженедельный отчет стейкхолдерам: статус, блокеры, риски, решения.
- Раз в две недели steering-встреча: решения по области, бюджету и срокам.

## Пакет отчетности
- Burn-up (объем поставки).
- Уверенность в контрольных точках (RAG-статус).
- Тренд дефектов.
- Процент прохождения тестов.
- Индекс стабильности окружения.

## Бюджет и загрузка
- Планирование по stream-aligned командам.
- Ограничители загрузки:
- максимум 75% плановой утилизации,
- 15% буфер на внеплановые работы,
- 10% на техдолг и усиление безопасности.

## Управление изменениями
- Изменение объема выше согласованного порога проходит через CCB (совет по управлению изменениями).
- Ускоренный процесс для изменений security/compliance.

## Шаблон для Confluence (еженедельный статус)
### 1. Сводка для руководства
- Общий статус (RAG)
- Ключевые успехи
- Ключевые проблемы

### 2. Статус поставки
- Прогресс целей спринта
- Статус контрольных точек

### 3. Риски/проблемы/зависимости
- Новые
- Эскалированные
- Закрытые

### 4. Качество
- Прогресс тестирования
- Тренд дефектов
- Статус гейтов

### 5. Финансы/загрузка
- План против факта
- Прогноз

### 6. Требуемые решения
- Владелец решения
- Срок решения

## PM-чеклист перед Go-Live
- Управленческие согласования завершены.
- Модель поддержки подтверждена.
- Runbook и процесс инцидентов подтверждены.
- Состав команды для hypercare утвержден.
- Пакет коммуникаций для стейкхолдеров опубликован.
