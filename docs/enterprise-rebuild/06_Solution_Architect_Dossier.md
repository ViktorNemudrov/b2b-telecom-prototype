# Досье для солюшн-архитектора

## Цель
- Спроектировать целевое решение продукта (бизнес + технологии + ограничения поставки).

## Архитектурное видение
- Канальная B2B-платформа самообслуживания со слоем оркестрации AI, изолированными доменами и управляемым риском генеративных ответов.

## Целевые способности
- Безопасная авторизация и сессия.
- Оркестрация AI с детерминированными защитными правилами.
- Доменные микросервисы/API-модули: счета, обращения, звонки.
- Единая наблюдаемость и управляемость.
- Контролируемая выкатка и фича-флаги.

## Ключевые архитектурные решения (черновик ADR)
- ADR-01: BFF/API Gateway слой между FE и внутренними сервисами.
- ADR-02: Оркестрация AI как отдельный backend-компонент (не только в клиенте).
- ADR-03: Детерминированные сценарии как движок политик поверх живого ИИ.
- ADR-04: Событийная телеметрия для продуктовых и технических метрик.
- ADR-05: Пошаговое выделение из монорепозитория в модульные ограниченные контексты.

## Референсная архитектура (логически)
- Слой пользовательского опыта: веб/PWA-клиенты.
- API/BFF-слой: сессия/авторизация, агрегация, оркестрация.
- Доменный слой: сервисы биллинга, тикетов, аналитики звонков.
- AI-слой: интенты, политики, маршрутизация моделей, проверки безопасности.
- Слой данных: операционная БД + аналитическое хранилище + кэш.
- Платформенный слой: CI/CD, мониторинг, секреты, инфраструктура как код.

## Стратегия перехода (Демо -> Продуктив)
- Трек A: стабилизация текущего FE с выносом бизнес-логики в backend.
- Трек B: ввод канонических контрактов данных/API.
- Трек C: поэтапная замена mock-данных через интеграционные адаптеры.
- Трек D: промышленное усиление (security/perf/ops).

## Анализ вариантов
- Вариант 1: полный greenfield-перезапуск (высокая стоимость, чистый дизайн).
- Вариант 2: strangler-подход поверх текущей кодовой базы (сбалансированный риск/сроки).
- Вариант 3: перенос демо с постепенными исправлениями (высокий операционный риск).
- Рекомендация: вариант 2 (strangler) с контролируемыми релизными волнами.

## Связка NFR/SLO
- SLO доступности по каждой способности.
- Бюджеты задержек по каждому пользовательскому пути.
- Бюджеты ошибок для AI/интеграций живого ИИ.
- Цели восстановления:
- RTO <= 60 min,
- RPO <= 15 min для критичных данных.

## Стратегия интеграций
- Anti-corruption layer для внешних billing/call/ticket систем.
- Contract-first с versioning и политикой обратной совместимости.
- Retry/backoff/idempotency для операций с финансовым влиянием.

## Безопасность и соответствие по принципу «по дизайну»
- Нулевая утечка секретов в клиентский бандл.
- Сервисные аккаунты по принципу наименьших привилегий.
- Политика минимизации и хранения данных.
- Аудитный след для чувствительных действий.

## Архитектурные риски
- Сильная связность UI и доменной логики.
- Непредсказуемость внешних LLM API.
- Сложность сквозной наблюдаемости между сервисами.
- Рост стоимости инференса без управленческого контроля.
